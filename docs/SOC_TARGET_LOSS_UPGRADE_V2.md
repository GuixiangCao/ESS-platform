# SOC目标损失算法升级说明

## 升级时间
2026-02-02

## 升级内容

### 旧逻辑（v1.0）
故障发生在某个充/放电周期时，只检查该周期结束时刻的SOC值。

**问题：**
- 如果故障发生在11:00-13:00充电周期，但13:00-15:00也是充电周期
- 旧逻辑只检查13:00的SOC
- 但实际应该检查15:00的SOC，因为那才是连续充电的极大值点

### 新逻辑（v2.0）
故障发生在某个充/放电周期时，**向后查找所有连续的相同类型周期**，在最后一个周期结束时检查SOC极值。

**改进：**
1. ✅ 自动识别连续的相同类型周期
2. ✅ 找到连续周期的最后一个周期结束时刻
3. ✅ 在极值点检查SOC（充电找极大值，放电找极小值）
4. ✅ 更准确地反映实际的充放电完成情况

## 对比示例

### 案例：电站231，2025-12-22，告警1

**充放电策略：**
```
00:00-08:00: 充电 (90kW)
08:00-11:00: 放电 (240kW)
11:00-13:00: 充电 (300kW)  ← 故障发生在这里（11:50）
13:00-15:00: 充电 (60kW)   ← 连续充电周期
15:00-20:00: 放电 (300kW)  ← 周期类型变化
```

**旧逻辑（v1.0）：**
- 检查时刻：13:00
- 检查范围：单个充电周期（11:00-13:00）
- 问题：忽略了13:00-15:00也是充电周期

**新逻辑（v2.0）：**
- 检查时刻：15:00 ✅
- 检查范围：连续充电周期（11:00-15:00，共2个周期）✅
- 说明：`连续充电周期11:00-15:00 (连续2个周期)结束时SOC=58.60%，未达到目标95%（极大值）`

**实际结果：**
- 15:00时SOC：58.60%
- 目标：95%
- 亏损电量：36.40%
- SOC目标损失：¥16.09

## 技术要点

### 1. 周期类型定义
- `ctype = 1`: 充电周期
- `ctype = 2`: 放电周期
- `ctype = 3`: 待机周期（不计算损失）

### 2. 连续周期判断
```javascript
// 判断下一个周期是否为相同类型
if (nextSlot.ctype === ctype) {
  // 相同类型，继续查找
  lastSameCycleIndex = i;
} else {
  // 不同类型，停止查找
  break;
}
```

### 3. 极值检查
- 充电周期：检查是否达到 **95% 极大值**
- 放电周期：检查是否达到 **8% 极小值**

### 4. 说明信息格式
```
连续充电周期11:00-15:00 (连续2个周期)结束时SOC=58.60%，未达到目标95%（极大值），亏损电量=36.40%
```

格式说明：
- `连续充电周期`: 周期类型
- `11:00-15:00`: 完整时间范围
- `(连续2个周期)`: 包含的周期数量
- `SOC=58.60%`: 实际SOC值
- `未达到目标95%（极大值）`: 目标值和极值类型
- `亏损电量=36.40%`: 差值百分比

## 向后兼容性

### 单个周期的情况
如果故障发生的周期后面没有连续的相同类型周期，逻辑与v1.0完全一致：

**示例：**
```
15:00-20:00: 放电 (300kW)  ← 故障发生在这里（15:39）
20:00-24:00: 待机 (0kW)    ← 类型变化
```

- v1.0检查时刻：20:00
- v2.0检查时刻：20:00 ✅（相同）
- 说明：`连续放电周期15:00-20:00结束时SOC=100.00%`

## 未来优化建议

1. **可配置的目标值**
   - 当前：充电95%，放电8%（硬编码）
   - 建议：允许用户为不同电站配置不同目标值

2. **SOC曲线分析**
   - 当前：只检查周期结束时刻的SOC
   - 建议：分析整个周期的SOC曲线，找到真实的极值点

3. **季节性调整**
   - 当前：全年使用固定目标值
   - 建议：根据季节调整目标SOC值（夏季可能不同于冬季）

4. **告警严重程度分级**
   - 当前：所有告警统一处理
   - 建议：根据告警严重程度应用不同的损失计算策略
